# Docker Guide - DRL gRPC Route Service

This document explains how the Docker image works and how to use it correctly as an external user.

## 1. What this image provides

The image runs a gRPC inference server that computes a route between two SUMO edge IDs.

Input:
- `start_edge` (first edge of the route)
- `dest_edge` (destination edge)

Output:
- Full ordered list of route edges generated by the DRL policy.

The service is implemented with:
- `grpc_server.py` (server entrypoint)
- `grpc_service.py` (gRPC service implementation)
- `route_engine.py` (policy + SUMO network inference)
- `coSim.proto` and generated protobuf stubs

Default model in the image:
- `results_full_objective/checkpoint_best`

## 2. High-level runtime flow

1. Container starts `grpc_server.py`.
2. gRPC server listens on port `50051` inside the container.
3. A client sends `GetAttribute` request with `attribute=ROUTE`.
4. Service extracts `start_edge` and `dest_edge` from `value.stringArray.values`.
5. `DRLRouteEngine` loads the checkpoint and computes the route.
6. Service returns the computed edge list in `value.stringArray.values`.

## 3. Prerequisites

- Docker Desktop or Docker Engine installed.
- A free host port (examples below use `50052` on host).
- If you want to use the included Python client outside Docker:
  - Python env with `grpcio` installed.

## 4. Build the image locally

From project root:

```bash
docker build -t drl-grpc-route:full-best .
```

## 5. Run the container

Recommended command:

```bash
docker run --rm --name drl_route_test --shm-size=2g -p 50052:50051 drl-grpc-route:full-best
```

Why `--shm-size=2g`:
- Ray can use shared memory during initialization.
- Larger `/dev/shm` avoids warnings and improves stability.

Port mapping meaning:
- Host `50052` -> Container `50051`.
- Your client must call `127.0.0.1:50052`.

## 6. API contract (important)

Service and method:
- `coSim.CoSim/GetAttribute`

Required request fields:
- `attribute = ROUTE`
- `type = GET`
- `id = any client vehicle/request id`
- `value.stringArray.values = [start_edge, dest_edge]`

Response:
- `value.stringArray.values = [edge_1, edge_2, ..., edge_n]`

Error semantics:
- `NOT_FOUND`: start/destination invalid or no route found.
- `INTERNAL`: engine/checkpoint/runtime failure.

## 7. Test method A - Included Python client (quickest)

From project root, in a second terminal (while container is running):

```bash
python test_client.py --host 127.0.0.1 --port 50052 --start-edge 361567664 --dest-edge 291922340#2
```

Expected:
- `RESPONSE RECEIVED FROM THE DRL SERVICE`
- Non-empty `Complete path` list.

## 8. Test method B - grpcurl without local install

If `grpcurl` is not installed on your machine, use the grpcurl Docker image.

### 8.1 Create request file

```powershell
@'
{
  "attribute": "ROUTE",
  "type": "GET",
  "id": "VEHICLE_TEST_01",
  "value": {
    "stringArray": {
      "values": ["361567664", "291922340#2"]
    }
  }
}
'@ | Set-Content request.json
```

### 8.2 Execute grpcurl in Docker

```powershell
docker run --rm -v "${PWD}:/protos" fullstorydev/grpcurl `
  -plaintext `
  -import-path /protos `
  -proto coSim.proto `
  -d @/protos/request.json `
  host.docker.internal:50052 coSim.CoSim/GetAttribute
```

Notes:
- `host.docker.internal` lets the grpcurl container call the service running on your host.
- Use `localhost:50052` only when grpcurl runs directly on host (not in container).

## 9. Choosing valid start/destination edges

The route engine expects edge IDs present in `sumo/Esch-Belval.net.xml`.

Known valid pair used in training and tests:
- Start: `361567664`
- Destination: `291922340#2`

If you provide invalid edges, you may get an empty route or `NOT_FOUND`.

## 10. Runtime options and customization

Container default command:

```bash
python grpc_server.py --checkpoint full_best --port 50051 --bind 0.0.0.0
```

You can override at `docker run` time:

```bash
docker run --rm --shm-size=2g -p 50052:50051 drl-grpc-route:full-best \
  python grpc_server.py --checkpoint full_best --port 50051 --bind 0.0.0.0
```

Checkpoint aliases currently supported by server:
- `full_best` -> `results_full_objective/checkpoint_best`
- `full_final` -> `results_full_objective/checkpoint`

## 11. Logs and health checks

Run container in foreground for immediate logs:

```bash
docker run --rm --shm-size=2g -p 50052:50051 drl-grpc-route:full-best

```

## 12. Common issues and fixes

### 12.1 `port is already allocated`

Cause:
- Host port already used.

Fix:
- Map another host port, for example `-p 50053:50051`.
- Then call client on `50053`.

### 12.2 `grpcurl is not recognized`

Cause:
- grpcurl not installed locally.

Fix:
- Use Test method B (grpcurl via Docker), or install grpcurl locally.

### 12.3 `Unable to start the Route Engine`

Possible causes:
- Wrong checkpoint path.
- Incompatible image/dependencies.
- Missing runtime files if image was modified.

Fix:
- Use the provided image and default `--checkpoint full_best`.
- Check container logs for exact stack trace.

### 12.4 Ray shared memory warning

Cause:
- `/dev/shm` too small.

Fix:
- Run with `--shm-size=2g` or higher.

## 13. Performance notes

- First request can be slower because policy and engine are lazily initialized.
- Subsequent requests are faster because engine stays in memory.
- Route length can be large depending on policy behavior.

## 14. Security and networking notes

- Service is plaintext gRPC (`insecure_channel`).
- For production or public exposure, put it behind secure infrastructure:
  - TLS termination
  - network ACLs
  - authenticated gateway


## 15. External user quick start

```bash
docker pull michelemarvulli/drl-grpc-route:full-best
docker run --rm --shm-size=2g -p 50051:50051 michelemarvulli/drl-grpc-route:full-best
```

Then send `GetAttribute` with:
- `attribute=ROUTE`
- `type=GET`
- `value.stringArray.values=[start_edge,dest_edge]`

Response returns the full route edge list.
